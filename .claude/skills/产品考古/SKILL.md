# 产品考古 SKILL

> 这是 PM AI 工作台最核心的 skill。系统化分析一个产品的过去、现在和未来。
> **核心铁律**：任何产品分析,必须先实地走查 + 真实用户声音 + 定价反推,不能只用搜索。

---

## 工作流总览

```
Phase 0: 澄清问题（3 个）
   ↓
Phase 1: 【强制】三件套实地调研(产品体验员 + 定价侦探 + 用户声音侦察员)
   ↓
Phase 2: 并行情报收集(市场研究员 + 竞品分析师 + 创始人调查员 + 趋势分析师)
   ↓
Phase 3: 数据综合(数据综合师)
   ↓
Phase 4: 四时空分析（前世/今生/未来/启发）
   ↓
Phase 5: 自动审查(怀疑论审查员 强制 4 题)
   ↓
Phase 6: 输出报告 + 保存到 outputs/
```

## Phase 0：澄清问题（必做，除非用户说"直接做"）

启动时先问以下 3 个问题（一次性问完，用编号列表）：

```
在开始考古之前，请回答 3 个问题，我会据此调整分析深度：

1. 你用这份分析做什么决策？
   A. 内部分享/写报告
   B. 决定要不要跟随这个产品方向
   C. 投资判断
   D. 求职面试准备
   E. 纯学习/好奇

2. 你的时间预算？
   A. 10 分钟速览（精简版）
   B. 30 分钟深度（标准版，推荐）
   C. 1 小时完整考古（深度版）

3. 你最关心哪个维度？（可多选）
   - 增长策略
   - 商业模式
   - 创始人决策
   - 技术演进
   - 监管/政策影响
   - 全部都要
```

收到回答后简短确认（1 句话），然后立即进入 Phase 1。**不要再追问其他问题。**

如果用户说"直接做"、"不用问"、"全都要"，使用默认配置：
- 决策类型：内部分享
- 深度：标准版（30 分钟）
- 维度：全部

## Phase 1：【强制】三件套实地调研(不能跳过)

> ⚠️ **这是本 SKILL 最关键的环节,跳过任何一项都会导致分析失真。**
> 之前 liblib.tv 分析的失败根因就是跳过了这一步——只看搜索结果不看真实产品。

### 必调三件套（并行执行）

```
@产品体验员 实地走查：{产品名} 官网/App
  - 抓取首页、定价页、活动页、注册流程页
  - 同时走查 1-2 个直接竞品做对比
  - 输出"5 个用户视角问题"的答案

@定价侦探 反推定价：{产品名}
  - 完整价格-额度-限制矩阵
  - 沉淀利润测算
  - 价格歧视手段识别
  - 真假补贴判断
  - 必须给出"反直觉判断"

@用户声音侦察员 挖掘真实用户反馈：{产品名}
  - Reddit / 小红书 / 知乎 / 即刻 / App Store 评论 / G2
  - Top 5 抱怨 + Top 3 称赞
  - 流失原因
  - 官方叙事 vs 用户现实的差距
```

### 完成检查清单(必须全部 ✓ 才能进入 Phase 2)

- [ ] 产品体验员是否实地访问过产品?(不是搜索描述)
- [ ] 产品体验员是否同时走查了 1-2 个竞品做对比?
- [ ] 定价侦探是否给出了沉淀利润测算?
- [ ] 定价侦探是否识别出了真假补贴?
- [ ] 用户声音侦察员是否拿到了至少 3 个社区的真实用户原话?
- [ ] 是否发现了至少 1 条反直觉判断?

**任何一项缺失 → 强制补做,不能用搜索结果替代。**

## Phase 4：四时空分析框架

### 数据标签规范（贯穿全报告）

每条关键信息必须用两组正交标签：

**第一组：确定性等级**
| 标签 | 标准 |
|---|---|
| 【事实】 | 多源印证 + 权威来源 |
| 【推断】 | 单源 + 合理逻辑推导 |
| 【假设】 | 无直接证据但逻辑合理 |

**第二组：来源立场**
| 标签 | 标准 | 示例 |
|---|---|---|
| 【官方自述】 | 来自公司自己的渠道 | 官方博客、财报中的战略描述、CEO 发言 |
| 【独立验证】 | 来自公司无法控制的独立渠道 | 第三方分析师、用户评论、竞品评价、监管方 |
| 【实地观察】 | 来自产品体验员实地走查 | 截图/页面交互观察 |
| 【用户原话】 | 来自用户声音侦察员 | Reddit/小红书/知乎用户原话 |

**使用规则**：
- 关键结论（如用户量、收入、市场份额）必须双标签标注
- 只有【官方自述】标签的核心数据，报告中必须注明"未经独立验证"
- 同一事实如果同时有官方自述和独立验证且一致 → 可信度最高
- 【实地观察】和【用户原话】优先级高于【官方自述】

### 硬数据处理规范（强制——所有量化数据必须遵守）

以下类型的数据属于"硬数据"，必须严格执行本规范：
- 财务类：ARR、收入、GMV、毛利、净利、ARPU、CAC、LTV
- 用户类：DAU、MAU、注册用户数、付费用户数、留存率
- 公司类：成立时间、融资轮次/金额/估值、团队人数
- 市场类：市场规模、市场份额、增长率

**处理规则**：

#### 规则 1：优先找原始出处
- 每个硬数据必须标注"数据来源 + 数据时间点"
- 优先级：财报/招股书 > 第三方审计 > 权威机构估算 > 媒体报道 > 官方宣称
- 找到出处的写法：`MAU 1.2 亿（来源：XX 2024 Q3 财报，截至 2024-09-30）`

#### 规则 2：找不到原始数据时，必须走推理链
如果无法直接找到某个硬数据，可以通过推理得出，但**必须完整展示推理过程**：

```
【推断】CAC ≈ $15
推理链：
1. 已知：2024 年营销费用 $3M（来源：XX 财报）
2. 已知：2024 年新增付费用户 20 万（来源：XX 财报）
3. 计算：$3M ÷ 20 万 = $15/用户
4. 局限：营销费用可能包含品牌支出，实际效果获客 CAC 可能更高
```

**推理链要素（缺一不可）**：
- 每一步的输入数据 + 出处
- 计算/推导逻辑
- 结论
- 局限性说明（哪些因素可能导致偏差？偏差方向？）

#### 规则 3：推理的"置信区间"
推理得出的数据必须给出范围而非精确数字：

| 置信度 | 写法 | 适用场景 |
|---|---|---|
| 高 | "约 X（±10%）" | 输入数据来自财报，推理步骤少 |
| 中 | "估计 X-Y 之间" | 输入数据来自第三方，推理步骤多 |
| 低 | "量级约 X，但不确定度高" | 输入数据本身是估算 |

#### 规则 4：绝对禁止
- ❌ 写一个精确数字但不标出处（如"CAC 为 $12"）
- ❌ 推理数据不展示推理过程
- ❌ 用模糊词代替数字（如"收入很高"、"增长快"）
- ❌ 把推理数据当事实引用（推理得出的数据在后续引用时仍须标【推断】）

### 【时空一】前世：产品起源与演进

必须回答以下问题：

| 问题 | 输出格式 |
|---|---|
| 诞生背景 | 200 字以内：解决什么问题？为什么是这个时间点？ |
| 创始团队最初的押注 | 1 句话核心假设 |
| 关键里程碑 | 时间线表格（3-5 个节点） |

| 重大转型 | 每次 pivot 的触发原因和效果 |
| 失败过的尝试 | 母公司之前的相关探索 + 教训 |

|---|---|---|---|---|
| | | | | |

- 反事实推断必须基于同时期其他公司的真实经历（不是凭空想象）
- 标注【推断】标签

| 时间 | 事件 | 影响级别 | 数据/证据 |

#### 商业模式深拆（必做,与定价侦探产出对应）

用 5 个问题穿透商业模式：

1. **谁付钱？**（用户付费 / 广告主 / 平台抽成 / 企业订阅 / 混合）
2. **付多少钱？**（ARPU、客单价、订阅金额）
3. **为什么付？**（核心价值主张，用户付费意愿来源）
4. **多久付一次？**（一次性 / 月度 / 年度 / 按使用量）
5. **毛利率级别？**（SaaS 70-90% / 电商 20-40% / 平台 30-50%）
6. **【新增】真实盈利暗线?**(沉淀利润、价格歧视、现金流暗线——必须引用定价侦探的产出)

#### Unit Economics 拆解（必做，即使是估算也要算）

| 指标 | 数值 | 来源/估算依据 | 标签 |
|---|---|---|---|
| CAC（获客成本） | | | 【事实/推断】 |
| LTV（用户生命周期价值） | | | |
| LTV / CAC | | | |
| Payback Period | | | |
| Gross Margin（毛利率,含沉淀） | | | |
| Gross Margin（毛利率,不含沉淀） | | | |
| Net Revenue Retention | | | |

**估算逻辑必须写出**：如"ARPU $X × 平均留存 Y 个月 = LTV $Z"。不写估算过程的数字不可接受。

#### 用户与场景（必做,引用用户声音侦察员的产出）

| 用户群体 | 占比估算 | 核心场景 | 付费意愿 | 真实吐槽 |
|---|---|---|---|---|
| | | | | |

#### 竞争格局（必做,需基于产品体验员的实地走查）

用"竞争象限图"思维：

| | 高功能完整度 | 低功能完整度 |
|---|---|---|
| **高细分聚焦** | （强专家型竞品） | （新进入者） |
| **低细分聚焦** | （现任领导者） | （边缘玩家） |

#### 价值网分析（必做——发现非显性竞争和依赖）

2×2 象限只能看到"直接同类"竞争者。价值网才能看到生态全局：

| 角色 | 具体玩家 | 与目标产品的关系 | 威胁/机会 |
|---|---|---|---|
| **互补品** | | 共生关系，谁强了我也强 | |
| **上游供应** | | 我依赖谁？他们卡我脖子吗？ | |
| **下游渠道** | | 我通过谁触达用户？ | |
| **替代品**（不同技术，同 JTBD） | | 用户可能用什么完全不同的方式解决同一问题？ | |
| **潜在颠覆者**（非显性赛道） | | 从意想不到的方向切入的威胁 | |

**关键追问**：目标产品最大的威胁是否来自"不在同一赛道内"的玩家？

### 【时空三】未来：3 年预判（强制三情景）

**这是最值钱的部分。** 必须给出三种情景：

#### 🟢 乐观情景（probability: ~25%）
- 触发条件：什么得发生？
- 增长路径：能到什么规模？
- 关键举措：他们会做什么？

#### 🟡 中性情景（probability: ~50%）
- 维持现有趋势会怎样？
- 主要约束是什么？

#### 🔴 悲观情景（probability: ~25%）
- 什么会让它衰落或崩盘？
- 警示信号是什么？

#### 关键变量监控清单

列出值得每月跟踪的 3-5 个信号：

| 信号 | 当前状态 | 警戒线 | 先行指标（提前 3-6 个月预警） | 监控方法 |
|---|---|---|---|---|
| | | | | |

**先行指标示例**（选最相关的填入）：
- GitHub star / commit 增速突变
- 招聘 JD 中关键词变化（如突然大量招"合规"岗）
- API 调用量 / SDK 下载量趋势
- 创始人公开发言频率和主题变化
- 竞品融资密度（同赛道 3 个月内多轮融资 = 赛道升温信号）
- 社交媒体讨论量拐点（从增长变平或下降）

#### 资源信号检查（重要!这是预测未来的最佳工具）

执行下列搜索并整理：

1. **招聘动向**（招聘 = 战略意图）
   - 搜索：`{公司} 招聘 LinkedIn` 或 `{公司} careers recent`
   - 关注：哪些岗位在猛招?(AI = 押注 AI;海外销售 = 国际化;合规 = 应对监管）

2. **收购/投资**
   - 搜索：`{公司} 收购 投资 最近`
   - 关注：买什么 = 缺什么

3. **专利申请**
   - 搜索：`{公司} 专利 patent 最近`

4. **高管公开发言主题**
   - 近 6 个月 CEO/CPO 在播客、访谈、财报会议上重复提到的关键词

5. **产品发布节奏**
   - 近 1 年的重大发布,看节奏(是加速还是减速?)

## Phase 5：自动审查（无需用户触发）

报告生成后，**自动执行以下两步**，不等用户指令：

### Step 1：自动调用 怀疑论审查员 压力测试(含强制 4 题)

```
@怀疑论审查员 审查以下分析：
[将 Phase 4 生成的完整报告传入]
```

怀疑论审查员会先回答强制 4 题:
- Q1: 看的是产品当下还是历史?(必须有产品体验员的走查)
- Q2: 看到的是官方话术还是用户真实体验?(必须有用户声音侦察员的产出)
- Q3: 赚钱的暗线是什么?(必须有定价侦探的反直觉判断)
- Q4: 钱从哪来?(LTV/CAC/Gross Margin 推算了吗?)

任何一题答"报告没覆盖"→ 强制重写对应章节。

### Step 1.5：根据 Phase 0 决策类型，自动追加 1 个视角审查

| Phase 0 用户选择的决策类型 | 自动追加的视角 | 审查重点 |
|---|---|---|
| B. 决定要不要跟随这个产品方向 | 审查 as 竞争对手 | 如果我是竞品，怎么攻击它？它的护城河真的存在吗？ |
| C. 投资判断 | 审查 as 投资人 | 退出路径？估值合理性？团队能否执行？ |
| D. 求职面试准备 | 审查 as 用户 | 真实用户体验如何？核心痛点解决了吗？ |
| A. 内部分享/E. 纯学习 | 审查 as 怀疑论者 | （已包含在 怀疑论审查员 中，可跳过此步） |

执行方式：用对应视角重新审视报告的核心结论（3-5 句话），输出补充洞察。

### Step 2：根据审查结果修订报告

- 将 怀疑论审查员 发现的问题分为"必须修复"和"补充说明"两类
- **必须修复**的问题（如数据错误、逻辑漏洞）：直接修订报告正文
- **补充说明**的问题（如信息缺口、反例）：在报告末尾新增「## 七、审查与局限性」章节，如实列出

### Step 3：提示用户可追加视角审查

修订后告诉用户：

```
报告已完成，已自动通过 怀疑论审查员 压力测试并修订。

如需追加特定视角审查,可输入:
- 「审查 as 投资人」—— 值不值得投?
- 「审查 as 竞争对手」—— 怎么攻击它?
- 「审查 as 用户」—— 真实用户怎么想?
```

## 反模式（绝对禁止）

❌ **跳过 Phase 1 三件套实地调研**(这是本次系统升级的核心动机)
   - 错误:直接用搜索结果写报告
   - 正确:先调用产品体验员实地访问 + 定价侦探反推 + 用户声音侦察员挖反馈

❌ **复述官方营销话术**
   - 错误:"抖音改变了内容消费方式"
   - 正确:"抖音 2018 年起靠两个机制实现 DAU 从 1 千万到 4 亿:① 推荐算法把内容分发去中心化 ② 创作者激励池让中尾部创作者也能赚钱"

❌ **模糊词汇回避结论**
   - 错误:"Notion 未来可能会面临一些挑战"
   - 正确:"Notion 在 2024 年遇到三个明确威胁:① Coda 被 Grammarly 收购整合 ② AI Notepad 类工具分流 ③ 企业市场被微软 Loop 压制"

❌ **数据无出处**
   - 错误:"用户量很大"
   - 正确:"MAU 1.2 亿(来源:母公司 2024 Q3 财报,截至 2024-09-30)"

❌ **没有 so what**
   - 错误:罗列一堆事实就结束
   - 正确:每个事实问"这对我意味着什么?"

❌ **编造信息**
   - 找不到数据时写 `[需补充:XX 数据]`
   - 推断必须标注 `【推断】`

❌ **不报告"反直觉判断"**
   - 一份没有反直觉判断的报告 = 一份白做的报告

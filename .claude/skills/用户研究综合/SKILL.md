# 用户研究综合 SKILL

> 哲学：研究的目的不是收集数据，而是产生洞察。
> 关键能力：把杂乱的原始素材，结构化为"可行动的发现"。

---

## 四步综合法

### Step 1：原子化（Atomization）

把所有素材打散成原子事实。

每个原子事实格式：

```
[原始引用] | [说话人 ID / 数据源] | [日期]
```

示例：
```
"我每次都要重新输入一遍地址，烦死了" | User-007（35 岁女性，月活跃用户）| 2026-04-15
"30% 用户在结账页面流失" | GA4 funnel data | 2026 Q1
```

**关键**：保留原话，不要过早概括。

### Step 2：聚类（Clustering）

把原子事实按主题归类。

常用聚类维度：
- **痛点类型**（功能缺陷 / 性能 / 易用性 / 设计 / 价格）
- **用户旅程阶段**（认知 / 注册 / 使用 / 付费 / 续费）
- **用户群体**（新手 / 高级用户 / 企业 / 个人）
- **情绪强度**（轻微抱怨 / 严重愤怒 / 流失警告）

### Step 3：证据强度评分

每个发现必须评分：

| 评级 | 标准 | 信心 |
|---|---|---|
| 🔴 强证据 | 3+ 独立用户提及 + 数据支持 | 90%+ 可信 |
| 🟡 中等证据 | 2-3 用户提及 OR 单一数据源 | 60-80% |
| 🟢 弱信号 | 1 用户 / 单一来源 | 40% 以下 |

**只把🔴和🟡 当作行动依据，🟢列为"待验证"。**

### Step 4：洞察生成

每个聚类升华为"洞察陈述"。

**好洞察的格式**：

```
[用户类型] 在 [场景] 因为 [根本原因] 而 [行为/反应]，
这导致了 [业务影响]。
```

示例：
```
新用户在首次注册时因为表单字段过多（10 个必填），
有 35% 在第 3 步放弃，
这导致每月损失约 800 个潜在付费用户（按 5% 付费率）。
```

**坏洞察**：
- "用户体验不好"（太泛）
- "应该改进 onboarding"（这是结论不是洞察）

## 并行处理多份访谈（Claude Code 特色）

如果用户给的是多个访谈 transcript 文件夹（10+ 份），使用并行处理：

```
@folder research/interviews/
对每份访谈，提取：
- 用户基本信息
- 5 个原子事实（带原话）
- 情绪关键词
- 反复出现的主题

每份生成 outputs/synthesis-{文件名}.md
最后我会综合所有 synthesis 生成总报告。
```

这样可以同时处理 10+ 访谈，速度提升 10 倍。

## 反模式

❌ **过早概括**：跳过原子化直接写"用户都觉得 onboarding 复杂"
❌ **代表性偏差**：只引用支持你假设的用户
❌ **混淆症状和原因**：用户说"想要功能 X"，但根本需求可能是别的
❌ **忽略沉默的多数**：访谈到的人 ≠ 全部用户
❌ **过度量化**：N=5 的访谈不能算 "60% 用户"

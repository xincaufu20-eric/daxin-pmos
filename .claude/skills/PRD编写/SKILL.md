# PRD 编写 SKILL

> 关键原则：永不盲生成(never generate blind)。先澄清,再动笔。

---

## 强制流程

### Step 1：先问 3-5 个澄清问题（绝对不能跳过）

不管用户多急，必须先问：

```
开始写 PRD 前，请先回答 3-5 个问题：

1. **核心假设**：你希望验证的假设是什么？（一句话）
   例："如果我们把 onboarding 从 5 步减到 2 步，Day 1 留存会提高 15%"

2. **目标用户**：这个功能服务谁？规模多大？
   - 我们已有 1000 万用户中的哪些子集？
   - 占比多少？

3. **问题严重程度**：用户现在怎么 workaround？
   - 有数据支持这是个真问题吗？
   - 不做会怎样？

4. **成功指标**：怎么算成功？
   - 主指标（必须移动）
   - 护栏指标（不能恶化）

5. **资源/时间**：什么时候需要？团队多少人？
```

如果信息不全，对每个不确定的地方，**用 `[需补充：……]` 标记**，而非编造。

### Step 2：用标准结构生成

```markdown
# PRD: {功能名}

> 作者：[用户]
> 日期：YYYY-MM-DD
> 状态：Draft / Review / Approved
> 目标发布：YYYY-MM-DD

## TL;DR
[3 句话：做什么、为谁、为什么]

## 1. 核心假设
我们相信 [X 用户] 在 [Y 场景] 会因为 [Z 价值] 而 [行动]，
我们将通过 [指标] 的 [幅度] 变化来验证。

## 2. 问题陈述
### 2.1 用户痛点
[用户在哪里、做什么、卡在哪里]

### 2.2 数据支持
- [数据 1：X% 用户在 step 3 流失]
- [数据 2：客服 ticket 中 Y% 与此相关]
- [需补充：具体 funnel 数据]

### 2.3 不做的代价
[预估损失：留存 / 收入 / 口碑]

## 3. 战略契合度
- 这与 OKR ___ 对齐
- 在产品战略中的优先级：P0 / P1 / P2

## 4. 解决方案
### 4.1 核心方案
[1 段话描述最核心的解决思路]

### 4.2 用户流程（before/after）
[用 Mermaid 图或文字描述]

### 4.3 关键功能点
1. [功能 1] - [一句话描述]
2. [功能 2]
3. [功能 3]

### 4.4 非目标（Non-goals）
明确不做什么：
- 我们 **不** 做 X，因为 ___
- 我们 **不** 做 Y，因为 ___

## 5. 成功指标
| 类型 | 指标 | 当前基线 | 目标值 | 测量方法 |
|---|---|---|---|---|
| 主指标 | | | | |
| 护栏 | | | | |
| 反指标 | | | | |

## 6. 用户故事
- As a [用户类型], I want to [动作], so that [结果]
- ...

## 7. 设计草图
[需补充：Figma 链接]

## 8. 技术考量
- 关键技术依赖
- 性能 / 扩展性 / 安全
- [需补充：详细技术方案 - 与 Eng Lead 讨论]

## 9. 风险与缓解
| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| | | | |

## 10. 时间线
- Discovery: 
- Design: 
- Eng: 
- QA: 
- Beta: 
- GA: 

## 11. Open Questions
- [ ] 问题 1
- [ ] 问题 2
```

### Step 3：自我审查（生成完主动建议）

```
PRD draft 完成。建议进行 3 次视角审查：

1. 「审查 as 工程师」—— 实现可行性、技术债、估时
2. 「审查 as 怀疑论者」—— 假设是否成立？数据可靠？反例？
3. 「审查 as 客户」—— 真的能解决我的问题？为什么不用现有方案？
```

---

## 保存

输出到 `outputs/PRD-{功能名}-{YYYY-MM-DD}.md`

如果用户要 Word 版本，调用 docx skill 生成 `.docx`。

---

## 反模式

❌ 不问问题直接写
❌ 编造数据
❌ 写超过 2 页（PRD 越长读者越少）
❌ 没有 non-goals
❌ 没有反指标
❌ 用模糊词如"提升体验"作为成功指标
